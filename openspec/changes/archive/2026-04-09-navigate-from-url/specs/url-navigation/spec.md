## ADDED Requirements

### Requirement: URL parsing for all ClickUp resource types
The system SHALL parse ClickUp web app URLs and extract resource identifiers. The parser SHALL support all URL patterns generated by the existing `ClickUpUrlGenerator`, including both long-form and short-form URLs. The parsed result SHALL be a typed enum distinguishing workspace, space, folder, list, task, comment, and document URLs.

#### Scenario: Parse workspace URL
- **WHEN** the system parses `https://app.clickup.com/workspace123`
- **THEN** the result is a workspace variant with `workspace_id = "workspace123"`

#### Scenario: Parse space URL (long-form)
- **WHEN** the system parses `https://app.clickup.com/ws123/v/o/s/space456`
- **THEN** the result is a space variant with `workspace_id = "ws123"` and `space_id = "space456"`

#### Scenario: Parse folder URL (long-form)
- **WHEN** the system parses `https://app.clickup.com/ws123/v/o/f/folder789`
- **THEN** the result is a folder variant with `workspace_id = "ws123"` and `folder_id = "folder789"`

#### Scenario: Parse list URL (long-form)
- **WHEN** the system parses `https://app.clickup.com/ws123/v/l/6-list012-1`
- **THEN** the result is a list variant with `workspace_id = "ws123"` and `list_id = "list012"`

#### Scenario: Parse task URL (short-form)
- **WHEN** the system parses `https://app.clickup.com/t/task345`
- **THEN** the result is a task variant with `task_id = "task345"`

#### Scenario: Parse comment URL (short-form with query parameter)
- **WHEN** the system parses `https://app.clickup.com/t/task345?comment=comment678`
- **THEN** the result is a comment variant with `task_id = "task345"` and `comment_id = "comment678"`

#### Scenario: Parse document URL (short-form)
- **WHEN** the system parses `https://app.clickup.com/d/doc901`
- **THEN** the result is a document variant with `doc_id = "doc901"`

#### Scenario: Reject non-ClickUp URLs
- **WHEN** the system parses `https://example.com/something`
- **THEN** the result is a parse error indicating the URL is not a recognized ClickUp URL

#### Scenario: Reject malformed URLs
- **WHEN** the system parses `not-a-url-at-all`
- **THEN** the result is a parse error indicating the input is not a valid URL

#### Scenario: Handle trailing slashes gracefully
- **WHEN** the system parses `https://app.clickup.com/t/task345/`
- **THEN** the result is a task variant with `task_id = "task345"` (trailing slash ignored)

### Requirement: URL input modal dialog
The system SHALL display a modal dialog when the URL navigation shortcut is triggered. The dialog SHALL accept text input via typing or clipboard paste, and SHALL support submission and cancellation via keyboard.

#### Scenario: Dialog displays on shortcut trigger
- **WHEN** the URL navigation shortcut is triggered
- **THEN** a centered modal dialog appears with the prompt "Enter ClickUp URL:"
- **AND** a single-line text input field is focused for immediate typing

#### Scenario: User submits URL with Enter
- **WHEN** the user has typed or pasted a URL into the dialog
- **AND** presses `Enter`
- **THEN** the dialog closes and the system attempts to parse and navigate to the URL

#### Scenario: User cancels with Escape
- **WHEN** the URL input dialog is visible
- **AND** the user presses `Esc`
- **THEN** the dialog closes without processing the input
- **AND** the previous screen state is restored

#### Scenario: User pastes URL with Ctrl+V
- **WHEN** the URL input dialog is visible
- **AND** the user presses `Ctrl+V`
- **THEN** the clipboard content is inserted into the text input field
- **AND** the cursor position updates to the end of the pasted text

#### Scenario: Empty submission shows inline error
- **WHEN** the user presses `Enter` with an empty input field
- **THEN** the dialog remains open
- **AND** an inline error message "Please enter a URL" is displayed

#### Scenario: Dialog input captures all keyboard events
- **WHEN** the URL input dialog is visible
- **AND** the user presses any key
- **THEN** the key is processed by the dialog (typed into the input field or handled as a shortcut)
- **AND** no underlying screen processes the key

### Requirement: Navigation to resource from parsed URL
The system SHALL navigate the user to the resource identified by a parsed URL. For resources requiring intermediate hierarchy levels (e.g., a task requires the list to be loaded), the system SHALL load each level sequentially before reaching the target.

#### Scenario: Navigate to workspace URL
- **WHEN** a workspace URL is parsed successfully
- **AND** the user is authenticated
- **THEN** the application navigates to the workspaces screen
- **AND** the target workspace is selected in the sidebar

#### Scenario: Navigate to list URL loads intermediate levels
- **WHEN** a list URL is parsed successfully (containing workspace and list IDs)
- **AND** the user is authenticated
- **THEN** the application navigates through spaces → folders → lists sequentially
- **AND** the target list is selected in the sidebar
- **AND** the task list view is displayed

#### Scenario: Navigate to task URL (short-form)
- **WHEN** a short-form task URL is parsed successfully (containing only task ID)
- **AND** the user is authenticated
- **THEN** the system fetches the task from the API
- **AND** navigates through the hierarchy (workspace → ... → list) using the task's metadata
- **AND** the task detail view is opened for the target task

#### Scenario: Navigate to comment URL
- **WHEN** a comment URL is parsed successfully (containing task and comment IDs)
- **AND** the user is authenticated
- **THEN** the system navigates to the task detail view
- **AND** the comment thread is loaded
- **AND** the target comment is highlighted or scrolled into view

#### Scenario: Navigate to document URL
- **WHEN** a document URL is parsed successfully
- **AND** the user is authenticated
- **THEN** the system fetches the document from the API
- **AND** opens the document view

#### Scenario: Navigate while already at target
- **WHEN** the user is already viewing the target resource
- **AND** navigates to the same resource via URL
- **THEN** the view is refreshed or reloaded
- **AND** no error is shown

### Requirement: Error handling for URL navigation failures
The system SHALL handle navigation failures gracefully, providing informative error messages and restoring the previous screen state when navigation cannot be completed.

#### Scenario: Invalid URL format shows inline error
- **WHEN** the user submits a URL that cannot be parsed
- **THEN** the dialog remains open
- **AND** an inline error "Unrecognized ClickUp URL format" is displayed
- **AND** the previous input text is preserved for correction

#### Scenario: Resource not found shows status bar error
- **WHEN** the URL parses successfully
- **AND** the API returns a 404 or similar error for the target resource
- **THEN** the dialog closes
- **AND** the status bar displays "Resource not found"
- **AND** the user remains on their previous screen

#### Scenario: Network error during navigation
- **WHEN** the URL parses successfully
- **AND** a network error occurs while loading an intermediate hierarchy level
- **THEN** the dialog closes
- **AND** the status bar displays "Navigation failed: <error>"
- **AND** the user remains on their previous screen

#### Scenario: Navigation without authentication
- **WHEN** the user triggers URL navigation on the auth screen (not authenticated)
- **THEN** the dialog does not open
- **AND** the status bar displays "Please authenticate first"

#### Scenario: API returns insufficient context for short-form URL
- **WHEN** a short-form task URL is parsed
- **AND** the API response does not include workspace or list IDs
- **THEN** the dialog closes
- **AND** the status bar displays "Cannot navigate: insufficient resource context"
- **AND** the user remains on their previous screen

### Requirement: URL parsing symmetry with URL generation
The system SHALL be able to parse any URL that it generates. For every URL produced by `ClickUpUrlGenerator`, the parser SHALL return the corresponding resource IDs that, if used to navigate, would reach the same resource.

#### Scenario: Round-trip for task URLs
- **WHEN** the system generates a task URL for task `task345`
- **AND** parses that same URL
- **THEN** the parsed result contains `task_id = "task345"`

#### Scenario: Round-trip for comment URLs
- **WHEN** the system generates a comment URL for task `task345`, comment `comment678`
- **AND** parses that same URL
- **THEN** the parsed result contains `task_id = "task345"` and `comment_id = "comment678"`

#### Scenario: Round-trip for document URLs
- **WHEN** the system generates a document URL for document `doc901`
- **AND** parses that same URL
- **THEN** the parsed result contains `doc_id = "doc901"`

### Requirement: Loading indicator during sequential navigation
The system SHALL display a loading indicator with progress information while navigating through hierarchy levels. The indicator SHALL show which level is currently being loaded.

#### Scenario: Loading indicator shows during multi-level navigation
- **WHEN** the system is navigating to a task URL that requires loading 4 hierarchy levels
- **THEN** the loading indicator displays "Loading... workspace" (or current level name)
- **AND** updates as each level completes

#### Scenario: Loading indicator clears on completion
- **WHEN** all hierarchy levels have been loaded
- **AND** the target resource is displayed
- **THEN** the loading indicator is cleared

#### Scenario: Loading indicator clears on failure
- **WHEN** navigation fails at any level
- **THEN** the loading indicator is cleared
- **AND** the error message is displayed

## MODIFIED Requirements

### Requirement: Keyboard shortcut chord support
The system SHALL support two-key chord shortcuts where the first key acts as a prefix leader. After the leader key is pressed, a brief pending state awaits the second key. If the second key matches the expected chord completion, the associated action is triggered. If a non-matching key is pressed, the second key is processed as normal input.

#### Scenario: g then u opens URL navigation
- **WHEN** the user presses `g`
- **AND** then presses `u` within the chord timeout
- **THEN** the URL input dialog opens

#### Scenario: g then non-u passes through second key
- **WHEN** the user presses `g`
- **AND** then presses `j` (not part of a `g` chord)
- **THEN** the `j` key is processed as normal navigation input (scroll down)

#### Scenario: Chord pending state times out
- **WHEN** the user presses `g`
- **AND** no second key is pressed within the chord timeout window
- **THEN** the pending state is cleared
- **AND** the next key is processed as normal input

#### Scenario: Esc cancels chord pending state
- **WHEN** the user presses `g`
- **AND** then presses `Esc`
- **THEN** the pending state is cleared
- **AND** no action is taken
